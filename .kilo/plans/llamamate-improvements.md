# LlamaMate — Improvements Plan

Covering 4 changes: rename to LlamaMate, ConfigurationManager refactor, log viewer, threads auto-detect.

---

## Change 1: Rename to "LlamaMate"

Rebrand the app from `llama-menubar` to `LlamaMate`.

### Files to modify (string changes only — no repo rename):

| File | What to change |
|------|---------------|
| `llama-menubar/build.sh` | `APP_NAME="llama-menubar"` → `APP_NAME="LlamaMate"`, VERSION line, Info.plist `CFBundleName`, `CFBundleIdentifier` (`com.llama.menubar` → `com.llamamate.app`) |
| `llama-menubar/Package.swift` | Target directory path `Sources/llmctl` stays (no change needed — SPM path is relative) |
| `llama-menubar/Sources/llmctl/main.swift` | `APP_SUPPORT_DIR` path (`llama-menubar` → `LlamaMate`), About dialog text, menu bar strings referencing "llama-menubar" |
| `llama-menubar/Sources/llmctl/ModelManager.swift` | Comment in generated server.conf: `generated by llama-menubar` → `generated by LlamaMate` |
| `llama-menubar/Sources/llmctl/SettingsWindowController.swift` | Window title, RAM check messages |
| `install-llama.sh` | RAM check echo messages: `[llama-menubar]` → `[LlamaMate]` |
| `release.sh` | Output filenames, DMG name |
| `README.md` | All `llama-menubar` references, Quick Start instructions, menu table, build instructions |
| `.app bundle dir` | `llama-menubar.app` → `LlamaMate.app` (in build.sh output) |

### Non-goals:
- GitHub repo rename (stays `llama.cpp_install_on_macos`)
- Directory restructuring (SPM target path stays `Sources/llmctl`)
- Breaking existing LaunchAgent or config paths (these reference `llama.cpp`, not the app)

## Change 2: ConfigurationManager

Eliminate duplicated config parsing between `ServerManager.loadConfig()` and `ModelManager.writeServerConfig()`.

### New struct `ServerConfig` (in `ConfigManager.swift`):

```swift
struct ServerConfig {
    var modelLabel = "", modelRepo = "", modelFile = "", modelPath = ""
    var port = "8080", context = "8192", host = "0.0.0.0"
    var ngl = "40", fa = "1", ctk = "q8_0", ctv = "q8_0"
    var threads = "0", batchSize = "512", profile = "balanced"
}
```

### New class `ConfigManager` (in `ConfigManager.swift`):

```swift
final class ConfigManager {
    static let shared = ConfigManager()
    
    func read() -> ServerConfig  // parses server.conf, returns struct with defaults
    func write(_ config: ServerConfig) throws  // writes server.conf
    func writeStartScript() throws  // generates llama-server-start.sh
}
```

### Changes to existing files:

| File | Action |
|------|--------|
| Create `ConfigManager.swift` | Single source of truth for config read/write + start script generation |
| `main.swift` — `ServerManager` | Replace inline parsing with `ConfigManager.shared.read()`. Remove duplicate `loadConfig()` internal logic, keep method as `loadConfig() { apply(config: ConfigManager.shared.read()) }` |
| `main.swift` — `ServerManager` | Remove `modelRepo`, `modelFile`, `host` individual setters (now part of `ServerConfig`) |
| `ModelManager.swift` | Replace `writeServerConfig()` with `ConfigManager.shared.write(config)`. Remove inline defaults. `saveSettings()` calls `ConfigManager.shared.write()` |
| `ModelManager.swift` | Remove `writeStartScript()` method (moved to `ConfigManager`) |
| `SettingsWindowController.swift` | Unchanged — already uses `ServerManager.shared` + `ModelManager.shared.saveSettings()` |

### `ConfigManager.writeStartScript()`:

Move the existing start script template from `ModelManager.writeStartScript()` into `ConfigManager`. Content identical (RAM check + all args). Called by:
- `ConfigManager.write()` (after writing server.conf)
- `ConfigManager.writeStartScript()` (standalone, for repair)

## Change 3: Log Viewer

Add a menu item + panel to view `~/Library/Logs/llama-server.log` and `llama-server.err.log`.

### New file or method:

Option A: Add to `ModelsWindowController.swift` (reuse pattern)
Option B: New small class `LogViewerController.swift`

Recommendation: **Option A** — add a `LogViewerController` new file, similar pattern to `SettingsWindowController`.

### Implementation:

```swift
final class LogViewerController: NSObject, NSWindowDelegate {
    static let shared = LogViewerController()
    
    private var window: NSPanel!
    private var logTextView: NSTextView!
    private var refreshTimer: Timer?
    
    func showWindow() { ... }
    
    @objc private func refresh() {
        let logPath = NSHomeDirectory() + "/Library/Logs/llama-server.log"
        let errPath = NSHomeDirectory() + "/Library/Logs/llama-server.err.log"
        let main = (try? String(contentsOfFile: logPath)) ?? "(no log yet)"
        let err = (try? String(contentsOfFile: errPath)) ?? ""
        logTextView.string = main + "\n\n--- stderr ---\n" + err
        logTextView.scrollToEndOfDocument(nil)
    }
}
```

### Menu item:

In `MenuBarController.refresh()`, add after "Server Settings…":

```swift
let logs = NSMenuItem(title: "View Server Logs…", action: #selector(openLogs), keyEquivalent: "")
logs.target = self
menu.addItem(logs)
```

### Window layout:
- NSPanel: 720×460, titled "Server Logs"
- NSTextView inside NSScrollView, monospaced font 10pt
- Non-editable, selectable
- Auto-refresh when opened (read files)
- Optional: Timer-based auto-refresh every 2 seconds while open

## Change 4: Threads Auto-Detect for Apple Silicon

Use `hw.perflevel0.logicalcpu` sysctl to detect P-core count on Apple Silicon and set optimal thread count.

### In `ServerManager.applyProfile()`:

After setting profile presets, if Apple Silicon and profile is "balanced":

```swift
if ServerManager.isAppleSilicon() && profile == "balanced" {
    var pcores: Int32 = 0
    var size = MemoryLayout<Int32>.size
    sysctlbyname("hw.perflevel0.logicalcpu", &pcores, &size, nil, 0)
    if pcores > 0 { threads = String(pcores) }
}
```

### In `SettingsWindowController`:

When displaying threads field, show hint with detected P-core count:

```swift
var pcores: Int32 = 0
var size = MemoryLayout<Int32>.size
sysctlbyname("hw.perflevel0.logicalcpu", &pcores, &size, nil, 0)
if pcores > 0 {
    hintLabel.stringValue = "0 = auto (\(pcores) P-cores detected)"
}
```

## Implementation Order

1. **ConfigurationManager** (refactor first — simplifies everything else)
2. **Threads auto-detect** (small change in ServerManager + SettingsWindowController)
3. **Log viewer** (new file + menu item)
4. **Rename to LlamaMate** (last — avoids touching files that are being refactored)

## Migration / Backward Compatibility

| Concern | Mitigation |
|---------|-----------|
| Old server.conf missing new fields | ConfigManager.read() applies defaults for missing fields |
| Existing LaunchAgent paths | Unchanged — LaunchAgent references `llama-server`, not the app |
| Existing ~/Library/Application Support/llama-menubar/ path on rename | App creates new dir, falls back to old if needed, or just uses new name going forward |
| Existing install-llama.sh bundled in old app | Re-bundled on next build |

## Risks

| Risk | Mitigation |
|------|-----------|
| Rename breaks path-dependent features | Only `APP_SUPPORT_DIR` changes — all other paths unchanged |
| ConfigManager refactor introduces regression | Same field names, same defaults — purely mechanical extraction |
| Log viewer reads large files | Use `NSString` with `NSData` reader, or limit to last N lines |
| P-core detection on Intel | `hw.perflevel0.logicalcpu` may return 0 on Intel; guard with `isAppleSilicon()` |
